# agent-config-skills

Four [Agent Skills](https://agentskills.io) that manage your AI coding setup — skills, guidance, and MCP servers — as **one personal git repo**, fanned out to every tool on every machine.

Supported fan-out targets: **Claude Code**, **Cursor**, and **Codex CLI** (project *and* user level), plus any of the 30+ agents [Ruler](https://github.com/intellectronica/ruler) can generate configs for at the project level.

## The problem

If you use more than one AI coding tool on more than one machine, your configuration fragments fast: Claude Code reads `~/.claude/` and `CLAUDE.md`, Cursor reads `.cursor/`, Codex reads `AGENTS.md` and `~/.codex/`, each keeps its own skills store and its own MCP registry — per machine. Skills drift between tools, guidance gets edited in one place and forgotten in three others, and a new laptop takes an afternoon to set up.

## The model

Everything you own lives **exactly once** in a private repo of yours (default `~/agent-config`):

```
agent-config/                  # your PRIVATE repo — the single source of truth
├── library/                   # every unit exists here exactly once
│   ├── skills/<name>/         #   Agent Skills (+ optional deps.toml per skill)
│   ├── guidance/*.md          #   reusable instruction fragments
│   └── mcp/servers.toml       #   catalog of every known MCP server block
├── global/                    # machine-wide scope: ruler.toml, AGENTS.md, skills -> library
├── projects/<repo>/           # one scope per project: ruler.toml, AGENTS.md, symlinks
└── bootstrap/                 # apply.sh (materialize + fan out), doctor.sh (deps)
```

Scopes *compose* from the library via symlinks; nothing is duplicated. Everything downstream is **disposable build output**, regenerated by `bootstrap/apply.sh`:

- **Project level** — each repo's `.ruler/` is a materialized copy of its scope; Ruler fans it out to `CLAUDE.md`, `.claude/skills/`, `.mcp.json`, `.cursor/*`, `.agents/skills/`, root `AGENTS.md`, `.codex/config.toml`, …
- **User level** — `~/.claude/CLAUDE.md`, `~/.codex/AGENTS.md`, `~/.claude/skills`, `~/.agents/skills`, and user-scope MCP registrations are regenerated from `global/` (marker-gated: files the system doesn't own are never overwritten).

Edit in the repo, push; other machines pull and regenerate. One writable source, everything else generated.

## The four skills

| Skill | What it does |
|---|---|
| `agent-config-init` | Entry point. Pull-first and idempotent: clones/pulls whatever exists, creates only what's missing, imports this machine's existing user-level config (with hash-based dedup), fans out, and offers to harvest existing repos. Safe to re-run anytime. |
| `agent-config-sync` | Daily driver. Pull → dependency doctor → re-materialize global + every wired project → clean stale generated skills. Push half publishes local edits, rescuing anything created in generated dirs first. |
| `agent-config-new-project` | Scaffold a new project's scope by shopping from your library — pick skills, guidance, MCP servers — then wire and verify the generated output matches the selection exactly. |
| `agent-config-harvest` | Onboard an existing repo with configs scattered across `.claude/`, `.cursor/`, `.kimi/`, etc.: inventory, content-hash dedup across stores, interactive conflict resolution, merge into a scope, cut over. Non-destructive until you approve. |

## Requirements

- An Agent Skills–compatible agent (built with/for Claude Code; the SKILL.md format is an open standard)
- [Ruler](https://github.com/intellectronica/ruler) (`npm install -g @intellectronica/ruler`, Node ≥ 20.19)
- `git`, `rsync` (preinstalled on macOS/Linux)

## Install

**Claude Code (as a plugin):**

```
/plugin marketplace add LightShards02/agent-config-skills
/plugin install agent-config-skills@agent-config-skills
```

**Any other Agent Skills–compatible tool** (Codex CLI, Cursor, …): copy the four skill directories into your agent's skills location (e.g. `~/.claude/skills/` or `~/.agents/skills/`).

Then just ask your agent:

> "Init my agent config on this machine."

`agent-config-init` takes it from there — including asking where your private config repo should live and importing what you already have. On later machines, the same request joins them to the existing repo instead. (After the first init, the four skills live in your library and propagate to your other machines automatically.)

## Design decisions worth knowing

- **Materialization, not live symlinks.** Ruler does not follow symlinked skill directories, so scopes are rsync-dereferenced (`-aL --delete`) into their consumption points before every apply. Symlinks stay in the repo (browsable, deduplicated); tools always read real files.
- **Dependencies are declared, not scripted.** A skill that needs a runtime carries a `deps.toml` (`runtimes = ["rg"]`, optional `brew = ["ripgrep"]`); MCP runtimes are derived from server `command` values. `doctor.sh` reports, and installs only with `--install` after approval. A pulled repo that auto-executes shell would be an attack surface — this one never does.
- **Pull-first, idempotent, convergent.** Every workflow fetches existing state before creating anything and treats "already exists" as a merge target. Re-running on a healthy machine changes nothing.
- **Additive-only MCP reconcile.** User-scope servers the repo doesn't list are never removed — they may be personal or app-managed.
- **Your repos stay yours.** Wiring uses `.git/info/exclude`, never the shared `.gitignore`; untracking previously-committed configs is offered with consequences explained, and only ever done with the repo owner's approval.

## Status

Validated by 34 sandboxed scenario runs across 3 iterations (fresh init, machine join, legacy imports, dirty-state safety, multi-store harvests, dependency flows, idempotency) plus a two-machine end-to-end over a real GitHub remote. ChatGPT desktop has no file-based config surface and is intentionally out of scope; Codex covers the OpenAI side.
